# Data Quality Summary

Цей документ підсумовує результати перевірок якості даних перед розрахунком `revenue`, `fuel cost allocation` та profitability KPI.

SQL-запити для перевірок збережені у файлі `sql/01_data_quality_checks.sql`.

---

## Мета перевірок

У сирих даних немає готових колонок `revenue` або `profit`. Ці метрики розраховуються аналітично через зв’язки між таблицями `trips`, `client_rates` та паливними даними.

Тому перед побудовою KPI-таблиць і Tableau dashboard датасет був перевірений на:

- покриття потрібного періоду;
- унікальність primary keys;
- пропуски в критичних полях;
- коректність зв’язків між таблицями;
- відповідність бізнес-правилам trucking-компанії;
- можливість коректного зіставлення поїздок із тарифами.

## Примітка щодо rate matching

`Revenue` розраховується через таблиці `trips` та `client_rates`.

Строга перевірка rate matching за клієнтом, датою, дистанцією та вагою виявила **701 нескасовану поїздку** без точного збігу з weight band.

Це не означає, що в даних відсутні тарифи. Причина полягає в тому, що на local routes фактичне завантаження (`cargo_tons_actual`) може перевищувати номінальну capacity трака та верхню межу тарифного weight band.

Для data quality було окремо перевірено наявність тарифу за логікою:

`client_id + date + distance`

Ця перевірка дала **0 failed records**, тобто всі нескасовані поїздки мають доступний тариф за клієнтом, датою та дистанцією.

Для розрахунку `revenue` у подальшому аналізі використовується ranked rate matching, який дозволяє вибрати найближчий доступний тарифний band, якщо точного збігу за вагою немає.

## Аналітичні рішення

| Ситуація | Рішення |
|---|---|
| `cancelled` trips | Виключаються з `revenue` та profitability KPI |
| `delayed` trips | Включаються в `revenue`, але аналізуються окремо як операційний ризик |
| Fuel purchases без `trip_id` | Fuel cost розподіляється на рівні `truck_id + month` |
| Profit на рівні окремої поїздки | Розглядається як estimated metric |
| Порожнє поле `clients.active_to` | Вважається коректним для постійних клієнтів |
| Strict weight-band gaps | Обробляються в revenue model через ranked rate matching |
